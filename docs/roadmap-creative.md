## 1. 产品层（直接可卖）

这些方向不需要 daudit 做任何新功能，用现有数据做不同呈现即可变现。

### 1.1 代码离岸检测

**一句话**：帮甲方定量评估外包团队交付质量。

**输入**：外包团队交付的 .pas 源码 + 内部自研代码参考基准

**输出**：

```
外包代码 vs 内部代码对比:
  命名规范违规率: 外部 23% vs 内部 5%
  死代码比例: 外部 15% vs 内部 3%
  注释率: 外部 2% vs 内部 15%
  隐晦写法: 外部频次是内部 6 倍
  总体评估: 建议要求返工
```

**定价**：按项目，5,000-20,000/次（远低于外包合同金额）

**依赖**：daudit --mode=audit 现有能力，加对比逻辑

---

### 1.2 新人入职加速包

**一句话**：给接手老项目的新人生成"项目生存手册"。

**输入**：项目 git 地址

**输出**：

```
你的新家: xxx.dproj（12 年历史，28 万行）

潜规则:
  · 全局变量 347 个
  · 以下 3 个函数叫 GetXxx，但实际会写入数据库（命名欺诈）
  · 那个叫 Utils.pas 的文件 2 万行，实际包含 6 个独立模块
  · 改这 3 个文件前请找对应的负责人（git blame）
```

**定价**：月付，按项目数

**依赖**：daudit + git blame 历史分析

---

### 1.3 老单体检中心

**一句话**：找出项目中多年没人敢碰的老代码，告诉团队每个文件该删还是该留。

**输入**：项目源码

**输出**：

```
老代码分析（5年以上未修改）:
  可直接删除（0 引用）: 8 个单元
  高危存活（仍在关键路径但写法过时）: 5 个单元
  需人工确认（通过 RTTI/动态调用无法追溯）: 3 个单元
```

**定价**：按次，或包含在企业订阅中

**依赖**：daudit 引用追踪（entity_refs 表）

---

### 1.4 技术债务征信报告

**一句话**：把审计结果做成像征信报告一样的格式，给管理层看。

**输入**：daudit audit JSON

**输出**：

```
─────────────────────────────────────
  Daofy 代码征信局 — 项目健康报告
─────────────────────────────────────

信用评分: 68/100（B 级） 较上期: +1

负面记录:
  资源泄漏高危: 12 处
  架构违规: 8 处
  过期依赖: 4 个

风险评估:
  当前: 中
  6 个月后: 中高（若不对 12 处资源泄漏采取措施）
─────────────────────────────────────
```

**定价**：免费版限 1 次/月，付费版不限 + 历史追踪

**依赖**：daudit audit 现有输出，仅需改报告模板

## 2. 传播层（引流传品）

这些方向做起来快、传播性强，目的是让更多 Delphi 开发者知道 daudit 的存在。

### 2.1 代码殡葬

**一句话**：评分太低时，生成一个带讽刺效果的"出殡页面"。

**触发**：`daudit --mode=audit` 评分低于 30

**输出**：

```
本项目代码质量评分: 23/100
剩余维护周期预测: 约 8 个月
—— Daofy 代码殡葬委员会

HTML 页面带落叶动画 + 死亡倒计时
```

**传播点**：截图发朋友圈/技术群，自动引流量

**工作量**：一个 HTML 模板 + 评分阈值逻辑

---

### 2.2 NFT 徽章

**一句话**：每个项目跑完审计获得一个质量评分徽章，可 mint 成链上凭证。

**触发**：每次 `--mode=audit` 自动生成 SVG

**输出**：

```
┌─────────────┐
│  DAOFY      │
│  评分 92/100 │
│  A+ 级      │
│  2026-05    │
└─────────────┘

SVG 可嵌入 README，也可上链 mint NFT
```

**传播点**：Delphi 老炮把金色徽章挂 LinkedIn，既是炫耀也是传播

**工作量**：SVG 生成 + 评分分级逻辑（几小时）

---

### 2.3 Delphi 社区排行榜

**一句话**：公开的 Delphi 项目质量排名，驱动开发者关注。

**触发**：每周扫描已知公开 Delphi 项目

**输出**：

```
 排名 | 项目 | 评分 | 代码量 | 违规密度
──────┼──────┼──────┼────────┼─────────
   1  | mormot2   | 94  | 280k  | 0.3/1k
   2  | delphier  | 91  | 45k   | 0.8/1k
   ...
```

**传播点**：开发者会自发查看排名、转发、争取上榜

**工作量**：一个静态站点或 GitHub Actions 定期跑 + 更新 README

---

### 2.4 代码时间线视频

**一句话**：用 Gource 风格 + daudit 语义分析染色，生成项目演进视频。

**触发**：`daudit --mode=timeline --output=evolution.mp4`

**输出**：60 秒 MP4，不同语义层（类/方法/违规）用颜色区分

**传播点**：短视频发 B 站/朋友圈，项目组全员转

**工作量**：中等，需视频生成管线

## 3. 生态层（建立壁垒）

这些方向需要更多开发投入，但一旦建成，竞争对手很难复制。

### 3.1 代码考古学家

**一句话**：对 git 历史跑语义分析，找出代码演化的因果链路。

**核心能力**：

```
daudit --mode=dig --commit=abc1234

考古发现:
  该提交改动: TStringList.Sort 前加了 OwnsObjects:=False
  动机推测: 典型的 Sort 触发比较器释放对象 bug
  当前状态: 后续已改为 TArray<T>，该行可安全删除
```

**收入模型**：作为 Daofy 企业版核心功能

**工作量**：大（需要 git 历史 + 语义分析的交叉推理）

---

### 3.2 架构警察

**一句话**：把团队架构规范固化为可执行的检查规则。

**核心能力**：

```json
{
  "rules": [
    "BLL 层不能引用 UI 单元",
    "所有 public 方法必须有异常处理",
    "DataAccess 类名必须以 TDAO 开头"
  ]
}
```

**收入模型**：规则模板市场（免费规则引流 + 企业规则收费）

**工作量**：中（规则引擎 + 可配置 DSL）

---

### 3.3 已离职专家知识抢救

**一句话**：识别只有特定开发者知道的隐式知识，防止人走茶凉。

**核心能力**：

```
王工（已离职 2 年）:
  他写的代码仍有 23 个单元在生产环境运行
  其中 5 个单元没有第二个人 commit 过
  热点: DataMigration.pas 的最后维护是他
```

**收入模型**：企业版，配合交接流程打包

**工作量**：中（git blame + 语义分析的交叉分析）

---

### 3.4 代码代谢率分析

**一句话**：追踪项目各模块的"生命体征"，识别危险区域。

**核心能力**：

```
代谢分析:
  高代谢区: PaymentGateway.pas（月变动 >20%，业务正常）
  低代谢高复杂度: LegacyOrderImport.pas（3 年未改，复杂度 92/100）
  代码体温: 43% 单元超过 6 个月无活跃修改
```

**收入模型**：企业版，持续监控仪表盘

**工作量**：中（需跨版本对比能力）

---

### 3.5 迁移适配器

**一句话**：告诉团队从旧 Delphi 版本升级需要改什么。

**核心能力**：

```
daudit --mode=compat D2007

兼容性问题列表:
  语法不兼容: 23 处（含已废弃的关键字）
  运行时行为差异: 7 处（Unicode/Ansi 转换、记录布局变化）
  自动可修复: 18 处（daudit 可直接生成 diff）
```

**收入模型**：按次收费，配合升级项目

**工作量**：大（需要多版本 Delphi 语法差异知识库）

---

### 3.6 新人入职加速

**一句话**：生成项目专属的"生存手册"，缩短新人上手时间。

**输出**：结合代码分析 + git blame + 架构分析的定制文档

**收入模型**：企业版订阅

**工作量**：中（主要是整合现有数据成可读文档）

---

### 3.7 团队基准

**一句话**：量化团队整体代码质量和一致性水平，给 Tech Lead 做管理依据。

**核心能力**：

```
团队分析:
  命名规范违规 23 处（80% 集中在同一人）
  方法长度标准差 28（差距极大）
  3 个核心模块异常处理覆盖率低于 60%
```

**收入模型**：SaaS 仪表盘，按团队大小收费

**工作量**：中（需按作者归因，git blame 已有）

---

### 3.8 代码能量计算器

**一句话**：把技术债务翻译成人民币，让管理层听懂。

**核心能力**：

```
技术债务估算: 340 人/天
年度修复成本: ¥480,000
12 个月后若不管: 债务升至 500 人/天
```

**收入模型**：企业订阅标配

**工作量**：小（基于现有违规数据 + 本地化薪资参数）

## 4. 实施建议

### 阶段一：传播先行（1-2 周）

这些方向成本最低、传播效果最强，先做：

```
 1. 代码殡葬        — 1 个 HTML 模板，改报告逻辑
 2. NFT 徽章         — SVG 生成，嵌入 README
 3. 技术债务征信报告  — 改审计报告模板
 4. Delphi 社区排行榜 — GitHub Actions + README
```

**目的**：让 Delphi 社区知道 daudit 存在。产品不需要完美，但要先有人用。

### 阶段二：产品验证（1-2 个月）

从"最容易让客户掏钱"的方向做起：

```
 5. 老单体检中心     — 引用追踪已有，包装成产品
 6. 新人入职加速     — git blame + 语义信息整合
 7. 代码离岸检测     — 对比逻辑 + 报价模板
```

**目的**：验证企业是否愿意为数据付费，而不只是为工具付费。

### 阶段三：生态壁垒（3-6 个月）

建立竞争对手难以复制的核心能力：

```
 8. 架构警察         — 可配置规则引擎
 9. 代码考古学家      — git 历史 + 语义交叉分析
10. 迁移适配器       — 多版本 Delphi 语法差异知识库
```

**目的**：让 daudit 不只是"另一个 linter"，而是 Delphi 生态的基础设施。

---

> 所有方向共享同一个底层：
> 
> `daudit.exe --mode ast` + `--mode audit` + `entity_refs 引用图谱`
> 
> 增加的功能绝大多数是呈现层，而不是引擎层。
> 先把引擎做扎实，后面每走一步都是增值。

### 5.7 定价原则

1. **个人版卖便利** — 一个人用，两顿饭钱，不纠结
2. **团队版卖数据** — 决策要用数据说话，你给他们数据
3. **企业版卖安全** — 出了问题有人负责，比什么都值钱
4. **免费层是获客成本** — LSP 免费是为了让所有 Delphi 开发者装上你家的东西

> 核心：Delphi 开发者已经习惯了高付费环境。你的工具比他们 IDE 便宜 30-100 倍，而且没有竞品。定价不需要激进，用户自然会比价。
